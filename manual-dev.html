Arquitetura de Dados

Fonte única de verdade — app.jsx

Todo o estado do app vive em app.jsx como useState. As páginas recebem dados via props e nunca acessam o storage diretamente.

// Estado global em app.jsx
const [pastos,   setP]  = useState([])
const [animais,  setA]  = useState([])
const [fin,      setF]  = useState([])
const [movs,     setMv] = useState([])
const [sal,      setSl] = useState([])
const [manejos,  setMj] = useState([])
const [adubacoes,setAdu]= useState([])

O padrão mk() — setter instrumentado

Toda mutação de estado usa um setter que automaticamente persiste no Supabase sem que as páginas precisem chamar save* manualmente:

// mk = make setter — grava no banco em cada mudança
const mk = (k, s) => fn => s(prev => {
  const next = typeof fn === 'function' ? fn(prev) : fn
  SAVERS[k]?.(next)   // ← fire-and-forget para Supabase
  return next
})

// Setters expostos para as páginas
const setPastos  = mk('pastos',  setP)
const setAnimais = mk('animais', setA)

Schema da tabela app_state

O Supabase usa uma única tabela key-value. Cada entidade é um jsonb completo:

CREATE TABLE app_state (
  key        text PRIMARY KEY,
  value      jsonb NOT NULL,
  updated_at timestamptz DEFAULT now()
);

-- Keys usadas:
-- 'pastos', 'animais', 'fin', 'movs',
-- 'sal', 'manejos', 'adubacoes', 'cfg'
ℹ️ Por que key-value? Permite evolução do schema sem migrações e mantém o app portável. A desvantagem é que arrays grandes são reescritos completos a cada save.

Fluxo de leitura na inicialização

Ao montar o app, useEffect carrega todos os dados em paralelo:

// Boot sequencial por ordem de dependência:

sal, manejos, adubacoes  → estado local

Storage — Supabase

Primitivas dbSet e dbLoad

dbSet usa upsert com onConflict: 'key' para criar ou atualizar atomicamente. É fire-and-forget — não bloqueia a UI:

export const dbSet = (k, v) => {
  supabaseClient
    .from('app_state')
    .upsert(
      { key: k, value: v, updated_at: new Date().toISOString() },
      { onConflict: 'key' }
    )
    .then(({ error }) => {
      if (error) console.error(`❌ dbSet [${k}]`, error)
      else       console.log(`💾 dbSet [${k}] OK`)
    })
}

RLS mínimo necessário

-- Executar no SQL Editor do Supabase:
ALTER TABLE app_state
  ENABLE ROW LEVEL SECURITY;

CREATE POLICY "anon_all" ON app_state
  FOR ALL USING (true)
  WITH CHECK (true);
⚠️ Produção: A política atual permite acesso anônimo total. Para múltiplos usuários, adicione coluna user_id e políticas por UUID de auth.

Variáveis de ambiente

Arquivo .env.local na raiz (nunca commitado):

VITE_SB_URL=https://xxxx.supabase.co
VITE_SB_ANON_KEY=eyJhbGci...

O prefixo VITE_ expõe as variáveis ao bundle do browser. Apenas a chave anônima pública vai aqui — nunca a service_role key.

Sistema de Tema

Como o tema funciona

O tema é um objeto plano com tokens de cor. O contexto React TC o distribui para todos os filhos. Cada componente faz const T = useT() e usa T.green, T.card, etc:

// constants.js
export const LIGHT = {
  green: '#1A7A4A', gDark: '#0F4D2E',
  bg: '#F5F5F7', card: '#FFFFFF',
  text: '#1D1D1F', gray: '#6E6E73',
  border: '#E5E5EA', shadow: 'rgba(0,0,0,0.06)',
  // … orange, red, blue, pink, purple, etc.
}

// app.jsx
<TC.Provider value={T}>{children}</TC.Provider>

Tokens principais

T.greenCor primária — botões e destaques
T.gDark / T.gLightVariações do verde para gradientes
T.gPaleFundo de cards informativos
T.bg / T.cardFundo de tela e fundo de card
T.text / T.grayTexto principal e texto secundário
T.border / T.shadowBordas e sombras

Algoritmos e Cálculos

Unidade Animal (UA)

// 1 UA = animal de 450 kg (padrão brasileiro)
ua = nAnimais × (pesoMedio / 450)

Recomendação de Adubação — calcRecomendacao()

Implementa o método Embrapa de saturação de bases:

// Calagem (NC em t/ha)
NC = CTC × (V2 − V1) / PRNT
// V2 = saturação alvo (70%), PRNT = 80 (cal padrão)

// Seleção de fórmula NPK:
if (P < 15) → '8-28-16'    // corrige fósforo
if (K < 70) → '20-05-20'   // corrige potássio
else        → '4-14-8'     // manutenção

// Cobertura nitrogenada:
ureiaKgHa = 100 // base
+20 se MO < 1.5 (solo muito pobre)
+10 se pH < 5.5 (acidez limita N)

Simulador de Confinamento

// Consumo diário: 2,5% do peso vivo médio
pesoMedio  = (pIni + pSaida) / 2
consDia    = pesoMedio × 0.025
consDiaLote = consDia × nAnimais
totalAlim  = consDiaLote × dias

// Ingredientes (proporções fixas Embrapa):
silagem      = 55%
fubaMillho   = 22%
farelSoja    = 13%
salProteínas =  7%
nucleoMin    =  3%

Cálculo de idade — calcIdade()

const dias  = diff em dias
const meses = Math.floor(dias / 30)
const anos  = Math.floor(dias / 365)

label:

Deploy e CI/CD

Fluxo completo de publicação

Mac/Terminal (~/Desktop/fazendinha)
  │
  ├── npm run dev         ← testa local em :5173
  │
  ├── git add .
  │   git commit -m "feat: ..."
  │   git push             ← envia para GitHub
  │
  └── Vercel detecta push ← build automático
      └── publica em ~60s  ← URL permanente

Variáveis de ambiente na Vercel

Configure no painel da Vercel (Settings → Environment Variables):

VITE_SB_URL        = https://xxxx.supabase.co
VITE_SB_ANON_KEY   = eyJhbGci...

O arquivo .env.local funciona só em desenvolvimento. Para produção, as variáveis devem ser cadastradas no painel da Vercel.
